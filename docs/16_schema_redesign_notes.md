# 스키마 재설계 메모

## 기본 정책

배움마루는 아직 실사용 데이터를 운영하지 않는다. 따라서 스키마가 목표 업무 모델을 잘못 표현하면 기존 DB 파일을 버리고 기준 스키마를 고친다. 호환 migration은 사용자가 기존 데이터 유지가 필요하다고 명시할 때만 작성한다.

## 현재 설계의 핵심

- 강좌는 `분야 → 과목 → 회차별 개설 → 복수 시간표`로 분리한다.
- 장소는 `건물 → 층 → 공간`과 `공간 ↔ 역할`을 분리한다.
- 신청은 회원과 강좌 개설의 관계이며, 상태 이력과 추첨 결과를 별도 보관한다.
- 추첨은 실행, 대상, 결과를 분리해 재추첨과 감사 가능성을 남긴다.
- 사용자/접속 코드/감사 로그/작업 이력은 운영 행위자를 추적한다.

## 현장 시간표와의 대응

예를 들어 `평생교육 - 한글교실 초급 - 20명 - 월/수 - 14:00~14:50 - 문화교육실(3층)`은 다음으로 저장한다.

```text
course_categories: 평생교육
courses: 한글교실 초급
terms: 2026년 2학기
course_offerings: 정원 20, 문화교육실, 접수 상태
course_schedules: 월 + 14:00~14:50, 수 + 14:00~14:50
```

## 다음 스키마 결정

`building_floors`를 실제 기준 데이터로 쓸지, 층 표시 문자열을 단순 보조 데이터로 쓸지 확정해야 한다.

현재 목표는 건물과 층을 한 번 등록해두고 공간에서 선택하는 경험이다. 따라서 다음 API/UI 전환 전에 `locations.building_floor_id` FK 도입 여부를 결정하고, 선택한다면 `floor_label` 중복 저장을 제거한다.

## UI 전환과 스키마

React 강좌 운영 화면은 정규화된 테이블을 사용자에게 그대로 노출하지 않는다. 사용자는 회차, 분야, 과목, 강사, 공간, 요일/시간대를 하나의 강좌 개설 폼에서 조합한다. API는 이 폼을 개설 command로 받아 필요한 기준 데이터 조회와 관계 저장을 수행한다.

이 원칙은 정규화와 좋은 입력 경험을 동시에 만족시키기 위한 것이다.
