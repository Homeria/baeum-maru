# 배움마루 개발 계획

## 1. 개발 원칙

- 포터블 노트북 모드를 먼저 완성한다.
- 상시 서버 모드는 나중에 확장한다.
- 핵심 업무 흐름을 먼저 완성한다.
- 처음부터 너무 많은 기능을 넣지 않는다.
- 미디어 업로드는 지원하지 않는다.
- 백업/복구는 MVP부터 포함한다.
- 실데이터 테스트 전까지는 더미 데이터만 사용한다.

## 2. 전체 개발 단계

```text
1단계: 프로젝트 뼈대
2단계: 런처 컨트롤 패널
3단계: 내장 웹서버
4단계: SQLite DB와 마이그레이션
5단계: 회원/강좌 관리
6단계: 수강신청 입력
7단계: 제한 규칙 검사
8단계: 추첨/대기자
9단계: 엑셀 입출력
10단계: 백업/복구
11단계: 접속자 상태 표시
12단계: 포터블 패키징
13단계: 실제 업무 시뮬레이션
```

## 3. 1단계: 프로젝트 뼈대

목표:

- Go module 생성
- 기본 디렉터리 구성
- README 작성
- 설정 로더 기본 구현
- 로그 초기화 기본 구현

산출물:

```text
go.mod
cmd/launcher/main.go
internal/config
internal/logging
README.md
```

## 4. 2단계: 런처 컨트롤 패널

목표:

- Fyne 창 실행
- 서버 상태 표시
- IP 목록 표시
- 포트 입력
- 버튼 배치
- 로그 표시 영역 추가

우선 구현 버튼:

- 서버 시작
- 서버 중지
- 관리 화면 열기
- 주소 복사
- 백업하기
- 백업 후 종료

아직 실제 서버 기능이 없어도 UI 이벤트와 상태 변화를 먼저 만든다.

## 5. 3단계: 내장 웹서버

목표:

- Go HTTP 서버 시작/중지
- `/` 기본 페이지
- `/admin` 관리자 페이지
- `/reception` 접수 페이지
- 브라우저 열기 기능
- graceful shutdown

확인할 것:

- 호스트 노트북에서 접속 가능
- 같은 내부망 다른 노트북에서 접속 가능
- Windows 방화벽 안내 필요 여부

## 6. 4단계: SQLite DB와 마이그레이션

목표:

- `data/center.db` 자동 생성
- SQLite 연결
- WAL 모드 적용
- foreign key 활성화
- busy timeout 설정
- 마이그레이션 실행

초기 테이블 후보:

```text
members
courses
registrations
lottery_results
waitlists
attendance
users
settings
audit_logs
schema_migrations
```

## 7. 5단계: 회원/강좌 관리

## 7.1 회원 관리

기능:

- 회원 등록
- 회원 검색
- 회원 수정
- 회원 상세 조회
- 중복 확인

## 7.2 강좌 관리

기능:

- 강좌 등록
- 강좌 수정
- 강좌 목록
- 강좌별 신청 현황 조회

이 단계의 목표는 수강신청에 필요한 기본 데이터를 넣을 수 있게 하는 것이다.

## 8. 6단계: 수강신청 입력

기능:

- 회원 검색
- 강좌 선택
- 신청 등록
- 신청 취소
- 회원별 신청 목록
- 강좌별 신청 목록

접수 화면은 빠르게 입력할 수 있어야 한다.

우선순위:

1. 회원 검색
2. 회원 선택
3. 강좌 선택
4. 신청 저장
5. 신청 목록 갱신

## 9. 7단계: 제한 규칙 검사

MVP 규칙:

- 동일 강좌 중복 신청 금지
- 1인 최대 신청 강좌 수 제한
- 동일 시간대 중복 신청 금지
- 접수 마감 후 수정 금지

추후 규칙:

- 1인 최대 당첨 강좌 수
- 강좌 분야별 제한
- 우선순위 대상자
- 반장 우선
- 기존 미수강자 우선
- 기관별 커스텀 룰

## 10. 8단계: 추첨/대기자

기능:

- 강좌별 신청자 수 확인
- 정원 초과 강좌 표시
- 추첨 실행
- 당첨자 저장
- 대기자 순번 생성
- 추첨 결과 조회
- 추첨 결과 로그 기록

주의:

- 추첨 전 백업 권장
- 추첨 중 신청 수정 잠금
- 중복 추첨 경고
- 결과 재생성 시 기존 결과 처리 정책 필요

## 11. 9단계: 엑셀 입출력

필수:

- 회원명단 가져오기
- 강좌목록 가져오기
- 신청 결과 내보내기
- 강좌별 수강자 명단 내보내기
- 추첨 결과 내보내기
- 대기자 명단 내보내기
- 출석부 내보내기

우선순위:

1. 엑셀 내보내기
2. 출석부 생성
3. 회원명단 가져오기
4. 강좌목록 가져오기

가져오기는 기관별 양식 차이가 크므로 MVP에서는 단순한 표준 양식부터 지원한다.

## 12. 10단계: 백업/복구

기능:

- `[백업하기]`
- `[백업 후 종료]`
- 최근 백업 시간 표시
- 백업 폴더 열기
- 백업 파일에서 복구

주의:

- 복구 전 현재 DB를 자동 백업한다.
- 실행 중 파일 단순 복사는 피한다.
- SQLite backup API 또는 안전한 백업 방식을 사용한다.

## 13. 11단계: 접속자 상태 표시

기능:

- 브라우저에서 heartbeat 전송
- 서버에서 접속 IP와 마지막 접속 시간 기록
- 컨트롤 패널에 접속자 수 표시
- 접속자 목록 표시

MVP 기준:

```text
최근 30초 이내 heartbeat가 있으면 접속 중
```

## 14. 12단계: 포터블 패키징

목표:

- Windows exe 빌드
- portable ZIP 생성
- 첫 실행 시 폴더 자동 생성
- README_처음사용.txt 포함
- config.json 기본값 포함

배포 구조:

```text
BaeumMaru_Portable_v0.1.0/
├─ BaeumMaru.exe
├─ README_처음사용.txt
└─ config.json
```

## 15. 13단계: 실제 업무 시뮬레이션

더미 데이터로 테스트한다.

테스트 시나리오:

- 회원 1,000명
- 강좌 50개
- 신청 3,000건
- 접수 노트북 2~4대 동시 접속
- 회원 검색
- 신청 등록
- 신청 취소
- 추첨 실행
- 결과 엑셀 출력
- 백업 후 종료
- 복구 테스트

## 16. 서버 모드 확장 계획

포터블 MVP 이후 진행한다.

기능:

- GUI 없는 server mode
- Linux binary
- Dockerfile
- 자동 백업
- 메일 알림
- PostgreSQL 지원
- systemd 서비스 예시
- 관리자 웹 설정 강화

## 17. 오픈소스 공개 전 체크리스트

- 실데이터 포함 금지
- 더미 데이터만 제공
- README에 개인정보 주의 문구 작성
- LICENSE 선택
- 설치/실행 방법 작성
- 스크린샷의 개인정보 제거
- 상표/이름 충돌 간단 확인
- config 예시에서 비밀번호 제거
- `.gitignore`에 runtime data 제외

## 18. Codex 작업 시 우선순위

Codex에 일을 시킬 때는 다음 순서가 좋다.

1. Go 프로젝트 기본 구조 생성
2. Fyne 런처 창 생성
3. 서버 시작/중지 기능 구현
4. SQLite 연결 및 마이그레이션
5. 회원/강좌 CRUD
6. 수강신청 입력
7. 추첨 로직
8. 엑셀 출력
9. 백업/복구
10. 포터블 빌드 스크립트

- 회원 등록/검색
- 강좌 등록

- 제한 규칙 검사
- 추첨 실행
- 결과 조회

- 백업 후 종료
- 백업 파일로 복구
