# 배움마루 MVP 범위 재정의

## 1. 재정의 목적

기존 문서의 MVP는 실제 업무 한 사이클을 거의 모두 포함한다. 방향은 좋지만 취미 사이드 프로젝트의 첫 구현 단위로는 크다.

이 문서는 배움마루를 다음 원칙으로 다시 쪼갠다.

- 빠르게 실행 가능한 데모를 먼저 만든다.
- 각 단계가 하나의 컴퓨터공학 주제를 학습하게 한다.
- `main` 브랜치에는 가능한 한 실행 가능한 상태를 유지한다.
- 실사용 가능한 완성도는 뒤 단계에서 점진적으로 올린다.

## 2. 단계 요약

```text
MVP-0: 로컬 웹 데모
MVP-1: 수강신청 업무 흐름
MVP-2: 추첨과 엑셀 출력
MVP-3: 포터블 실행, 백업, 복구, 패키징
MVP-4: 실제 업무 시뮬레이션
```

현재 코드는 MVP-0부터 MVP-3까지의 핵심 업무 흐름과 기본 로그인 보호를 대부분 구현했다. 다만 사용성, 실제 업무량 시뮬레이션, 고급 권한 체계는 아직 별도 브랜치로 닫아야 한다.

## 3. MVP-0: 로컬 웹 데모

### 목표

한 대의 개발 PC에서 서버를 실행하고, 브라우저로 접속해 최소 업무 데이터를 다룰 수 있게 한다.

### 포함 범위

- Go module 생성
- 기본 디렉터리 구조 생성
- 설정 로더
- 파일 로깅
- HTTP 서버 시작
- SQLite 연결
- 마이그레이션 실행
- 기본 웹 레이아웃
- 회원 등록/검색
- 강좌 등록/목록
- 회원을 강좌에 신청
- 강좌별 신청자 목록 조회

### 제외 범위

- Fyne 런처
- 내부망 접속자 상태 표시
- 추첨
- 엑셀 출력
- 백업/복구
- 로그인 고도화
- 포터블 ZIP 패키징

### 완료 기준

```text
go run ./cmd/server
```

위 명령으로 서버가 실행되고, 브라우저에서 다음이 가능하면 완료로 본다.

- `/` 접속
- 회원 1명 등록
- 강좌 1개 등록
- 해당 회원을 해당 강좌에 신청
- 강좌별 신청자 목록 확인

## 4. MVP-1: 수강신청 업무 흐름

### 목표

접수 직원이 실제 접수 업무를 흉내 낼 수 있는 흐름을 만든다.

### 포함 범위

- 관리자 화면과 접수 화면 분리
- 회원 검색 속도 개선
- 강좌 선택 UI
- 신청 등록/취소
- 회원별 신청 목록
- 강좌별 신청 목록
- 접수 기간 상태
- 신청 제한 규칙 1차 적용

### 필수 규칙

- 같은 회원은 같은 강좌에 중복 신청할 수 없다.
- 같은 회원은 같은 시간대 강좌에 중복 신청할 수 없다.
- 1인 최대 신청 강좌 수를 넘을 수 없다.
- 접수 마감 후에는 신청을 수정할 수 없다.

### 완료 기준

더미 데이터 기준으로 다음 시나리오가 가능하면 완료로 본다.

- 회원 100명 등록
- 강좌 10개 등록
- 신청 200건 입력
- 중복 신청이 차단됨
- 시간 중복 신청이 차단됨
- 접수 마감 후 신청 수정이 차단됨

## 5. MVP-2: 추첨과 엑셀 출력

### 목표

접수 이후 결과 정리 업무까지 연결한다.

### 포함 범위

- 정원 초과 강좌 표시
- 추첨 대상 강좌 선택
- 자동 추첨
- 당첨자 저장
- 대기자 순번 생성
- 탈락자 저장
- 추첨 실행 로그
- 추첨 seed 저장
- 결과 조회
- 엑셀 내보내기
- 출석부 엑셀 생성

### 완료 기준

다음이 가능하면 완료로 본다.

- 정원 이하 강좌는 자동 확정 처리
- 정원 초과 강좌는 추첨 처리
- 동일 seed로 추첨 결과를 설명할 수 있음
- 강좌별 당첨자, 대기자, 탈락자 목록 조회
- 결과를 엑셀 파일로 저장
- 출석부를 엑셀 파일로 저장

## 6. MVP-3: 런처, 백업, 포터블 패키징

### 목표

개발자가 아닌 담당자도 실행할 수 있는 포터블 프로그램 형태로 만든다.

### 포함 범위

- 콘솔형 포터블 런처
- 실행 시 서버 시작
- 실행 시 브라우저 열기
- 터미널 종료 또는 신호 수신 시 graceful shutdown
- 관리자 화면 열기
- 접수 화면 열기
- 수동 백업
- 백업 파일에서 복구
- Windows exe 빌드
- portable ZIP 생성

Fyne 기반 네이티브 런처 컨트롤 패널은 MVP-3의 필수 범위에서 제외하고, 사용성 개선 또는 배포 polish 단계에서 다시 검토한다.

### 완료 기준

압축을 푼 뒤 다음 흐름이 가능하면 완료로 본다.

```text
baeum-maru.exe 실행
→ 서버 자동 시작
→ 브라우저에서 접속
→ 신청 데이터 입력
→ 웹 관리자 화면에서 백업
→ 프로그램 종료 후 재실행
→ 재실행 후 데이터 유지 확인
→ 백업 파일 복구 확인
```

## 7. MVP-4: 실제 업무 시뮬레이션

### 목표

실제 기관 업무량에 가까운 더미 데이터로 안정성을 확인한다.

### 기준 데이터

- 회원 1,000명
- 강좌 50개
- 신청 3,000건
- 접수 브라우저 2~4개 동시 접속

### 확인 항목

- 회원 검색이 체감상 빠른가
- 신청 저장이 즉시 처리되는가
- 중복/시간 충돌 규칙이 정확한가
- 추첨 결과가 저장되는가
- 엑셀 출력이 수초 안에 끝나는가
- 백업/복구가 안정적인가
- 로그에 개인정보가 과도하게 남지 않는가

## 8. 우선순위 원칙

기능 우선순위는 다음 순서를 따른다.

1. 데이터가 정확히 저장된다.
2. 사용자가 실수하기 어렵다.
3. 문제가 생겼을 때 복구할 수 있다.
4. 화면이 빠르게 동작한다.
5. 포터블 배포가 쉽다.
6. 나중에 서버 모드로 확장할 수 있다.
