# 배움마루 업무 규칙

## 1. 규칙 설계 원칙

- 사용자가 이해할 수 있는 규칙명으로 관리한다.
- DB 제약과 서비스 계층 검사를 함께 사용한다.
- 규칙 위반 메시지는 접수 직원이 바로 이해할 수 있어야 한다.
- 추첨 결과는 재현 가능성을 고려한다.
- 예외 처리는 관리자만 수행할 수 있게 한다.

## 2. 회차 상태 규칙

### 상태

```text
draft: 준비 중
open: 접수 중
closed: 접수 마감
finalized: 결과 확정
```

### 허용 행동

| 행동 | draft | open | closed | finalized |
|---|---:|---:|---:|---:|
| 회원 등록/수정 | 가능 | 가능 | 가능 | 가능 |
| 강좌 등록/수정 | 가능 | 제한 | 제한 | 제한 |
| 신청 등록 | 불가 | 가능 | 불가 | 불가 |
| 신청 취소 | 불가 | 가능 | 관리자만 | 관리자만 |
| 추첨 실행 | 불가 | 불가 | 가능 | 제한 |
| 엑셀 출력 | 가능 | 가능 | 가능 | 가능 |

## 3. 신청 등록 규칙

### R-REG-001 같은 강좌 중복 신청 금지

한 회원은 같은 강좌 개설 건에 한 번만 신청할 수 있다.

DB 제약:

```text
UNIQUE(member_id, offering_id)
```

사용자 메시지:

```text
이미 신청한 강좌입니다.
```

### R-REG-002 취소된 신청 재신청

취소된 신청을 다시 신청하는 경우 새 행을 만들지 않고 기존 신청 상태를 `applied`로 되돌리는 방식을 우선 고려한다.

이유:

- 신청 이력을 추적하기 쉽다.
- `UNIQUE(member_id, offering_id)` 제약을 유지할 수 있다.

### R-REG-003 1인 최대 신청 강좌 수

회차별 설정값을 기준으로 회원의 활성 신청 수를 제한한다.

활성 신청 상태:

```text
applied
selected
waitlisted
confirmed
```

제외 상태:

```text
cancelled
rejected
```

설정값이 0이면 제한 없음으로 본다.

### R-REG-004 동일 시간대 중복 신청 금지

같은 회원은 시간이 겹치는 강좌를 동시에 신청할 수 없다.

시간 겹침 조건:

```text
same weekday
AND existing.start_time < new.end_time
AND new.start_time < existing.end_time
```

사용자 메시지:

```text
같은 시간대의 다른 강좌를 이미 신청했습니다.
```

### R-REG-005 접수 마감 후 신청 수정 금지

회차 상태가 `closed` 또는 `finalized`이면 일반 접수 화면에서 신청 등록, 수정, 취소를 막는다.

관리자 예외:

- 잘못 입력한 데이터 정정
- 민원 대응
- 추첨 전 오류 수정

관리자 예외는 감사 로그를 남긴다.

## 4. 강좌 정원 규칙

### R-CAP-001 정원 초과 신청 허용

배움마루는 접수 단계에서 정원 초과 신청을 허용한다.

이유:

- 많은 기관이 선착순이 아니라 추첨제로 운영한다.
- 정원 초과 여부는 추첨 단계에서 처리한다.

### R-CAP-002 정원 이하 강좌 자동 확정

접수 마감 시 신청 수가 정원 이하인 강좌는 추첨 없이 `selected` 또는 `confirmed` 처리할 수 있다.

초기 정책:

```text
정원 이하 -> selected
정원 초과 -> lottery 대상
```

`confirmed`는 담당자가 최종 확정하는 상태로 남겨둔다.

## 5. 추첨 규칙

### R-LOT-001 추첨 대상

다음 조건을 만족하는 강좌만 추첨 대상이다.

- 회차 상태가 `closed`
- 강좌 상태가 `closed` 또는 접수 종료 상태
- 활성 신청 수가 정원보다 큼
- 아직 완료된 추첨 결과가 없거나 관리자가 재추첨을 명시함

### R-LOT-002 추첨 전 백업 권장

추첨 실행 전 백업을 권장한다. MVP-3부터는 추첨 실행 버튼 근처에 백업 버튼을 제공한다.

### R-LOT-003 seed 저장

추첨 실행 시 seed를 저장한다.

목적:

- 결과 설명 가능성
- 테스트 재현성
- 버그 추적

### R-LOT-004 대기자 순번

정원 초과 강좌에서 당첨되지 않은 신청자는 추첨 순서에 따라 대기 순번을 부여한다.

예:

```text
정원 3명, 신청 6명
1~3번째: selected
4번째: waitlisted, result_order = 1
5번째: waitlisted, result_order = 2
6번째: waitlisted, result_order = 3
```

초기 MVP에서는 `rejected`와 `waitlisted`를 모두 쓰지 않고, 정원 초과 탈락자를 전부 `waitlisted`로 볼 수 있다. 기관 정책상 대기자를 운영하지 않는 경우에만 `rejected`를 사용한다.

### R-LOT-005 재추첨 정책

같은 강좌에 이미 완료된 추첨 결과가 있으면 기본적으로 재추첨을 막는다.

재추첨이 필요한 경우:

- 관리자 확인을 받는다.
- 기존 결과를 삭제하지 않고 새 `lottery_run`을 만든다.
- 현재 유효한 결과가 무엇인지 표시한다.

MVP에서는 단순화를 위해 재추첨을 막는 것으로 시작한다.

## 6. 상태 전이

### 신청 상태 전이

```text
applied -> cancelled
applied -> selected
applied -> waitlisted
applied -> rejected
selected -> confirmed
selected -> cancelled
waitlisted -> selected
waitlisted -> cancelled
```

MVP에서 우선 필요한 전이:

```text
applied -> cancelled
applied -> selected
applied -> waitlisted
selected -> confirmed
```

## 7. 엑셀 입출력 규칙

### 가져오기

- 표준 양식만 먼저 지원한다.
- 필수 컬럼이 없으면 가져오기를 중단한다.
- 일부 행 오류는 오류 목록으로 보여준다.
- 업로드 원본은 영구 저장하지 않는다.

### 내보내기

- 내보낸 파일은 `exports/`에 저장한다.
- 파일명에는 회차, 출력 종류, 날짜를 포함한다.
- 개인정보가 포함된 파일임을 README와 화면에 안내한다.

예:

```text
2026_summer_lottery_results_2026-07-01_1730.xlsx
```

## 8. 백업/복구 규칙

### 백업

- 수동 백업은 언제든 가능하다.
- 실행 중 백업은 SQLite backup API 또는 안전한 방식으로 수행한다.
- 백업 성공 후 파일 존재와 크기를 확인한다.

### 복구

- 복구 전 현재 DB를 자동 백업한다.
- 복구는 서버 중지 상태에서 수행한다.
- 복구 후 DB 연결과 마이그레이션 상태를 확인한다.

## 9. 감사 로그 규칙

반드시 감사 로그를 남길 행동:

- 로그인 성공/실패
- 회원 등록/수정
- 강좌 등록/수정
- 신청 등록/취소
- 관리자 예외 수정
- 추첨 실행
- 백업/복구
- 설정 변경

로그에 남기지 않을 것:

- 전화번호 전체
- 생년월일 전체
- 민감한 비고 내용
- 비밀번호 또는 해시 원문
