# 화면 흐름

## 화면 분리 원칙

- 호스트 런처는 실행 PC의 서버와 운영 환경을 제어한다.
- 업무 웹은 접수 직원과 업무 관리자가 반복적으로 사용하는 화면이다.
- 호스트 런처는 pywebview 독립 창으로만 제공하며 LAN 사용자에게 노출하지 않는다.
- 같은 데이터를 여러 화면에서 보여주더라도 각 화면의 주된 작업을 하나로 유지한다.

## pywebview 호스트 런처

```text
대시보드
├─ 업무 서버 시작/중지/재시작
├─ bind mode, port, 내부망 접속 주소
└─ DB/백업/최근 오류 요약
접속 관리
├─ 접속 코드 발급
├─ 사용 가능/만료/폐기 목록
└─ 코드 원문/상세, 연장, 폐기, 목록 숨김
초기 환경
├─ 기관 정보와 로고
├─ 건물/층/공간/역할
└─ 신청 개수 등 운영 기본값
로그
설정
백업/실험실
```

런처 내부 탭을 이동해도 입력과 선택 상태를 유지한다. 애플리케이션 실행 시 독립 런처 창은 열지만 업무 서버를 자동으로 시작하지 않는다.

## 직원 업무 웹

```text
로그인
├─ 접수
│  ├─ 회원 검색 또는 신규 회원 입력
│  ├─ 선택 회원의 기존 신청 요약
│  ├─ 희망 강좌 다중 선택
│  └─ 한 번의 검증/저장
├─ 회원
│  ├─ 검색/필터/목록
│  ├─ 상세와 수정
│  └─ 신청 이력
├─ 강좌 운영
│  ├─ 회차/분류/과목/강사/공간 기준 데이터
│  └─ 강좌 개설과 복수 시간표
├─ 신청 현황
├─ 추첨
├─ 출력
└─ 설정/감사 로그
```

## 접수 흐름

1. 직원이 회원번호, 이름, 연락처 일부로 검색한다.
2. 기존 회원을 선택하거나 새 회원 정보를 입력한다.
3. 여러 강좌를 고른다. 선택 즉시 시간 충돌, 중복, 접수 상태, 정원 정보를 보여준다.
4. 저장 시 service가 같은 업무 규칙을 다시 검증한다.
5. 성공하면 접수 use case를 소유한 Repository가 회원 변경과 여러 신청을 하나의 transaction으로 반영한다.
6. WebSocket 이벤트를 받은 다른 브라우저는 관련 query만 다시 읽는다.

## 충돌과 갱신 UX

- 목록 화면은 관련 event가 오면 query를 다시 읽는다.
- 폼 편집 중에는 자동 reload하지 않고 새 데이터 안내를 보인다.
- 저장 충돌은 최신 조건과 다시 시도할 수 있는 경로를 함께 보여준다.
- 추첨, Excel, 복구처럼 큰 변경은 진행 상태와 잠금 이유를 명확히 표시한다.
- 서버 연결이 끊기면 입력을 잃지 않고 재연결 상태를 보여준다.
